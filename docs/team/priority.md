# Issue 优先级分级（草案）

> **尚未定稿**。由主维护者牵头，Issue 分诊人与领域负责人讨论后更新本页。定稿前可参考下列描述打标签：`priority:p0` / `priority:p1` / `priority:p2`。

## P0 — 必须尽快处理

**定义**：影响用户**正常训练或启动**，或存在**数据/发布安全事故**风险；应在当前或下一个可发布版本中修复。

**示例（非穷举）**：

- 参数正确仍「无法连接后端」、训练无法启动或立即退出
- 整合包缺目录导致 `/api/run` 500（如 `config/autosave`）
- 训练监控 / TensorBoard 入口错连，导致用户完全无法使用监控
- Release 7z 带入维护者本机私有配置或用户数据
- 契约路径被误改导致整合包 / AutoDL 无法启动

**不适用 P0**：纯 UI 不好看、文档笔误、未影响主流程的优化。

## P1 — 重要，排入近期版本

**定义**：不阻断核心训练，但明显损害体验、可维护性或多环境一致性；或团队已达成共识的中等规模改造。

**示例**：

- 端口治理 P0 小改（服务清单、启动日志、AutoDL 严格端口等）
- 监控页误导性错误提示（训练仍在进行但显示致命错误）
- 整合包缺静态资源、主页链接错误
- 某优化器 / 精度组合导致可复现的 `loss=nan`（需调查与缓解）
- 设计草案中已共识、已拆 Issue 的落地项

## P2 — 可排队、可分批

**定义**：增强、重构、长期架构、插件体系；或依赖面大、可先用兼容方案顶住的项。

**示例**：

- 端口规范全文落地（281xx 内部端口池、完整健康检查、插件注册）
- 新功能探索、前端大改
- 文档全面翻新、自动化矩阵一次性补齐
- 社区运营类（回复 SLA 等，当前用户量较少可暂缓）

## 使用方式

1. Issue 分诊时打标签（需在仓库中创建对应 labels）。
2. 发版前由主维护者确认：**本版是否包含未关闭的 P0**。
3. 领域负责人在自己范围内可建议调整优先级，**项目级调整由主维护者决定**。
